STANDARDS OVERVIEW
================================================================================

✅ Code Style
   • Formatter: Ruff (88 char lines, 4-space indent)
   • Naming: PascalCase (classes), snake_case (functions), UPPER_SNAKE_CASE (constants)
   • Docstrings: Google style
   • Imports: Organized by stdlib → third-party → local

✅ Type Safety
   • Full type annotations on all code (100% mypy/pyright coverage target)
   • Python 3.10+ syntax (list[T], str | None, not List, Optional)
   • Pydantic v2 for data validation at system boundaries
   • No Any without justification

✅ Architecture
   • Project layout: src/ directory structure (Python Packaging Authority)
   • Single Responsibility Principle per class
   • Dependency injection (no globals)
   • Domain-specific exception hierarchies
   • Standard logging module (never print())

✅ Version Control
   • Git Flow: main/develop with feature/bugfix/hotfix branches
   • Conventional Commits: feat/fix/docs/style/refactor/perf/test/chore
   • Atomic commits: one logical change per commit
   • Present tense messages: "add feature" not "added feature"

✅ Pull Requests
   • One feature/fix per PR (max ~400 lines)
   • Clear description + test plan
   • Require code review before merge
   • Squash or rebase for clean history

✅ Testing
   • Framework: pytest with conftest.py fixtures
   • Coverage: 80% minimum (100% for critical code)
   • Organization: unit, integration, E2E tests
   • FIRST principles: Fast, Independent, Repeatable, Self-checking, Timely
   • Mock external dependencies only

✅ Code Review
   • Correctness: logic, edge cases, error handling
   • Security: SQL injection, hardcoded secrets, XSS, input validation
   • Testability: code can be tested, fixtures work
   • Performance: no N+1 queries, reasonable memory usage
   • Maintainability: clear naming, short functions, low coupling
